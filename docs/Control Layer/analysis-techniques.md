# Claude Code 분석 기법 운용 가이드

## 개요

코드 작성, 리뷰, 디버깅, 설계 과정에서 사용하는 분석 기법 모음.
각 기법은 특정 상황에서 활성화하며, PDCA 사이클에 매핑하여 운용한다.

---

## 분석 기법 목록

### 1. 딥리서치 (Deep Research)

**목적:** 결과의 원인을 깊이 추적하여 근본 원인을 파악한다.

**활성화 시점:**
- 버그의 근본 원인을 찾을 때
- 성능 저하의 원인을 분석할 때
- 에러 로그를 추적할 때
- 기존 코드의 동작 원리를 파악할 때

**프롬프트 형식:**
```
딥리서치: [대상]의 [현상]에 대해 근본 원인을 분석해.
- 현재 상태: [증상 설명]
- 기대 상태: [정상 동작]
- 관련 코드: [파일/함수명]
```

**출력 기대값:**
- 원인 체인 (A → B → C → 근본 원인)
- 관련 코드 라인 특정
- 수정 방안 제시

**PDCA 단계:** Check

---

### 2. 딥시뮬레이션 (Deep Simulation)

**목적:** 구현 전에 결과를 예측하고 사이드이펙트를 사전에 파악한다.

**활성화 시점:**
- 새 기능 구현 전
- DB 스키마 변경 전
- API 설계 결정 전
- 아키텍처 변경 전

**프롬프트 형식:**
```
딥시뮬레이션: [변경 사항]을 구현했을 때 예상되는 결과를 시뮬레이션해.
- 변경 내용: [구체적 변경]
- 현재 시스템: [영향 받을 수 있는 부분]
- 시뮬레이션 범위: [정상 케이스 / 에러 케이스 / 극단적 케이스]
```

**출력 기대값:**
- 정상 시나리오 결과
- 실패 시나리오와 조건
- 사이드이펙트 목록
- 리스크 등급 (상/중/하)

**PDCA 단계:** Plan

---

### 3. 레드팀 (Red Team)

**목적:** 공격자 관점에서 코드의 보안 취약점을 찾는다.

**활성화 시점:**
- API 엔드포인트 작성 후
- 인증/결제 로직 완성 후
- 고객 입력을 받는 부분 완성 후
- 외부 서비스 연동 후

**프롬프트 형식:**
```
레드팀: [대상 코드/API]를 악의적 사용자 입장에서 분석해.
- 대상: [엔드포인트/함수/페이지]
- 공격 시나리오: 최소 5개 도출
- 각 시나리오별 심각도와 방어법 제시
```

**출력 기대값:**
- 공격 벡터 목록 (인젝션, 인증 우회, 권한 상승 등)
- 각 공격의 심각도 (Critical / High / Medium / Low)
- 구체적 방어 코드 또는 설정

**PDCA 단계:** Check

---

### 4. 엣지케이스 스윕 (Edge Case Sweep)

**목적:** 기능의 극단적 상황을 전수 조사하여 누락된 처리를 찾는다.

**활성화 시점:**
- 기능 구현 완료 후, 테스트 전
- 사용자 입력을 받는 모든 부분
- 파일 업로드/다운로드 처리
- 외부 API 호출 부분

**프롬프트 형식:**
```
엣지케이스 스윕: [기능명]에서 발생 가능한 극단적 상황을 전부 나열해.
- 대상 함수/컴포넌트: [이름]
- 입력 유형: [사용자 입력 / 파일 / API 응답 등]
- 각 케이스별 현재 처리 여부와 필요한 대응 제시
```

**출력 기대값:**
- 엣지케이스 전체 목록
- 현재 처리됨 / 미처리 구분
- 미처리 항목의 구현 방안

**PDCA 단계:** Do

---

### 5. 아키텍트 리뷰 (Architect Review)

**목적:** 시니어 아키텍트 관점에서 설계의 확장성, 성능, 유지보수성을 평가한다.

**활성화 시점:**
- 새로운 기능 설계 완료 후
- DB 스키마 확정 전
- API 구조 결정 전
- 시스템 구성 변경 전

**프롬프트 형식:**
```
아키텍트 리뷰: [설계/스키마/API 구조]를 시니어 아키텍트 관점에서 리뷰해.
- 평가 기준: 확장성 / 성능 / 유지보수성 / 보안
- 현재 설계: [구조 설명 또는 코드]
- 예상 규모: [사용자 수 / 데이터량 / 트래픽]
```

**출력 기대값:**
- 각 평가 기준별 점수 또는 등급
- 문제점과 개선안
- 우선순위가 매겨진 액션 아이템

**PDCA 단계:** Plan

---

### 6. 디버그 트레이스 (Debug Trace)

**목적:** 에러 발생 과정을 한 줄 한 줄 추적하여 정확한 발생 지점을 특정한다.

**활성화 시점:**
- 원인 모를 버그
- 간헐적으로 발생하는 문제
- 딥리서치로 원인 범위를 좁힌 후 정밀 추적

**프롬프트 형식:**
```
디버그 트레이스: [에러/현상]의 실행 흐름을 한 단계씩 추적해.
- 시작점: [요청 진입 / 함수 호출 / 이벤트 발생]
- 끝점: [에러 발생 / 잘못된 출력]
- 각 단계에서 변수 상태와 분기 조건을 표시
```

**출력 기대값:**
- 실행 흐름 단계별 나열
- 각 단계의 입력/출력 값
- 분기점에서의 조건 평가 결과
- 문제 발생 정확한 라인

**PDCA 단계:** Check

---

### 7. 유저 스토리 워크스루 (User Story Walkthrough)

**목적:** 실제 사용자 입장에서 기능을 처음부터 끝까지 사용해보며 UX 문제를 찾는다.

**활성화 시점:**
- UI/UX 완성 후
- 고객 접점 기능 배포 전
- 신규 플로우 설계 검증 시

**프롬프트 형식:**
```
유저 워크스루: [사용자 유형]이 [목표]를 달성하는 전체 과정을 시뮬레이션해.
- 사용자: [신규 고객 / 재방문 / 관리자 등]
- 시작: [진입점]
- 목표: [달성하려는 것]
- 매 단계에서 불편한 점, 막히는 점, 이탈 가능 지점을 짚어줘
```

**출력 기대값:**
- 단계별 사용자 행동과 시스템 응답
- 이탈 위험 지점 표시
- UX 개선 제안

**PDCA 단계:** Plan

---

### 8. 디펜던시 임팩트 (Dependency Impact)

**목적:** 특정 변경이 시스템의 다른 부분에 미치는 영향을 사전에 파악한다.

**활성화 시점:**
- DB 스키마 변경 전
- API 응답 형식 변경 전
- 공통 컴포넌트/유틸 수정 전
- 환경변수/설정 변경 전

**프롬프트 형식:**
```
디펜던시 임팩트: [변경 대상]을 수정할 때 영향 받는 범위를 전부 찾아줘.
- 변경 내용: [구체적 변경]
- 영향 받는 것 / 안 받는 것 구분
- 수정 필요한 파일과 라인 특정
```

**출력 기대값:**
- 영향 받는 파일/함수/컴포넌트 목록
- 각각 수정 필요 여부
- 수정 순서 (의존 관계 고려)

**PDCA 단계:** Do

---

### 9. 벤치마크 시나리오 (Benchmark Scenario)

**목적:** 시스템이 실제 트래픽과 부하를 감당할 수 있는지 시뮬레이션한다.

**활성화 시점:**
- 배포 전
- 성능 이슈 의심 시
- 인프라 스펙 결정 시

**프롬프트 형식:**
```
벤치마크 시나리오: [시스템]이 다음 조건에서 버틸 수 있는지 분석해.
- 트래픽: [동시 접속 / 일일 요청 수]
- 데이터: [DB 크기 / 파일 수]
- 하드웨어: [서버 스펙]
- 병목 구간과 한계점을 찾아줘
```

**출력 기대값:**
- 각 구간별 예상 응답 시간
- 병목 구간 특정
- 한계 트래픽 (이 이상이면 장애)
- 스케일업/스케일아웃 방안

**PDCA 단계:** Check

---

### 10. 리팩터 브리프 (Refactor Brief)

**목적:** 코드 리팩토링의 우선순위와 전략을 수립한다.

**활성화 시점:**
- 코드 복잡도가 높아졌을 때
- 기능 추가가 어려워졌을 때
- 버전 업그레이드 전

**프롬프트 형식:**
```
리팩터 브리프: [대상 코드]의 리팩토링 우선순위를 정해줘.
- 대상: [파일/모듈/컴포넌트]
- 현재 문제: [복잡도 / 중복 / 결합도 등]
- 1순위~5순위로 뭘 먼저 해야 하는지
- 각각 난이도, 예상 시간, 이유 제시
```

**출력 기대값:**
- 우선순위 목록
- 각 항목의 난이도/시간/효과
- 리팩토링 전후 구조 비교

**PDCA 단계:** Act

## 조합 운용 패턴

### 패턴 A: 신규 기능 개발
```
1. 딥시뮬레이션 → 구현 전 리스크 파악
2. 아키텍트 리뷰 → 설계 검증
3. (구현)
4. 엣지케이스 스윕 → 극단적 상황 처리
5. 레드팀 → 보안 점검
6. 유저 워크스루 → UX 검증
```

### 패턴 B: 버그 수정
```
1. 딥리서치 → 원인 범위 좁히기
2. 디버그 트레이스 → 정확한 지점 특정
3. 딥시뮬레이션 → 수정안의 사이드이펙트 예측
4. 디펜던시 임팩트 → 수정 영향 범위 확인
5. (수정)
```

### 패턴 C: 리팩토링
```
1. 리팩터 브리프 → 우선순위 수립
2. 디펜던시 임팩트 → 변경 영향 범위 파악
3. 딥시뮬레이션 → 리팩토링 후 예상 결과
4. (리팩토링)
5. 벤치마크 시나리오 → 성능 퇴행 없는지 확인
```

### 패턴 D: 배포 전 점검
```
1. 엣지케이스 스윕 → 누락된 처리 확인
2. 레드팀 → 보안 취약점 점검
3. 벤치마크 시나리오 → 부하 테스트
4. 유저 워크스루 → 최종 UX 확인
```

## 검증 전략: 위험 등급별 최소 수행 기준

세션 시작 시 작업의 위험 등급을 먼저 판단하고, 등급에 따라 최소 검증 횟수를 적용한다.
모든 기법은 최소 2회 수행을 기본으로 하며, 에이전트 병렬 운용으로 효율을 확보한다.

**세션 시작 프롬프트:** "이번 작업은 [작업 내용]. 위험 등급 판단하고 검증 계획 세워."

---

### 에이전트 × 기법 담당 매핑

| 에이전트 | 담당 기법 | 역할 |
|----------|-----------|------|
| 기획자 | 딥시뮬레이션, 유저 워크스루 | 구현 전 예측, UX 검증 |
| 디렉터 | 아키텍트 리뷰 | 설계 품질 판단, 방향 결정 |
| 빌더 | 엣지케이스, 디펜던시 | 구현 중 극단 상황 처리, 영향 범위 확인 |
| 리뷰어 | 딥리서치 | 결과 원인 분석, 근본 원인 추적 |
| 테스터 | 레드팀, 벤치마크, 디버그 트레이스 | 보안 점검, 부하 테스트, 실행 추적 |
| 리팩터 | 리팩터 브리프 | 코드 개선 우선순위 수립 |
| 스키머 | 사전 스캔 (코드/문서 훑기) | 작업 전 현황 파악, 컨텍스트 수집 |
| 크롤러 | 외부 정보 수집 | 시세/정가/레퍼런스 데이터 수집 |

### 에이전트 풀 운용 (총 8개 / Max 플랜 기준)

```
전체 에이전트 풀:
  빌더, 디렉터, 리팩터, 리뷰어, 테스터, 스키머, 기획자, 크롤러

등급별 가동/대기:

등급 1: 가동 1~2개 / 대기 6~7개
  → 빌더만으로 충분
  → 나머지 전원 대기

등급 2: 가동 4개 / 대기 4개
  → 기획자 + 빌더 + 테스터 + 디렉터
  → 리뷰어, 리팩터, 스키머, 크롤러 대기

등급 3: 가동 6개 / 대기 2개
  → 기획자 + 디렉터 + 빌더 + 테스터 + 리뷰어 + (필요 시 스키머)
  → 리팩터, 크롤러 대기

등급 4: 가동 8개 전원 / 대기 0개
  → 전 에이전트 투입
  → 크롤러는 외부 레퍼런스 수집 지원
```

---

### 등급 1 (저위험): 최소 4회 / 가동 1~2개 / 대기 6~7개

**해당 작업:**
- 버튼 색상/크기 변경
- 텍스트 문구 수정
- CSS 스타일링 조정
- 콘솔 로그 추가/제거

## 운용 원칙

1. **위험 등급 우선**: 세션 시작 시 반드시 위험 등급을 먼저 판단한다.
2. **최소 2회 원칙**: 모든 기법은 기법당 최소 2회 수행한다. 1회 수행으로 통과 판정하지 않는다.
3. **병렬 우선**: 같은 Phase 내 에이전트는 병렬로 동시 수행하여 시간을 절감한다.
4. **Phase 순서 준수**: Phase A(사전 분석) → Phase B(구현 후 검증) → Phase C(종합 판정) 순서를 지킨다.
5. **상향 적용**: 판단이 애매하면 한 등급 올려서 적용한다.
6. **결과 기록**: 분석 결과를 tasks.md 또는 context.md에 기록하여 추적한다.
7. **추가 검증**: 문제 발견 시 최소 기준과 무관하게 해결될 때까지 추가 검증한다.
8. **디렉터 최종 승인**: 등급 3 이상은 디렉터가 Phase C에서 최종 승인해야 완료된다.
