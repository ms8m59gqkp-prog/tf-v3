## 1) Phase 0 선행 체크 (DDL/제약/인덱스/RLS 변경 시 필수)

### 1.1 Preflight (데이터 정리 런북 준수)
- [ ] UNIQUE/제약 추가 전 **중복 탐지 쿼리 실행**했고 결과를 저장했다
- [ ] 고아 FK 탐지(해당 시) 쿼리 실행했고 결과를 저장했다
- [ ] 정리 정책(살릴 row 기준) 문서화했다
- [ ] 정리 SQL은 **수동 승인 후에만** 실행한다(자동화 금지)

### 1.2 마이그레이션 안전성
- [ ] `CONCURRENTLY` 가능한 인덱스는 CONCURRENTLY로 생성한다
- [ ] 다운타임/락 리스크(ALTER TABLE 큰 변경) 평가했고 순서를 정했다
- [ ] 롤백 전략이 있다(제약 제거/정책 롤백/함수 이전 버전)

## 2) RLS/공개 경계 심사 (Public/anon 관련 변경 시 필수)

### 2.1 정책 최소권한(특히 orders)
- [ ] `USING (true)` 같은 **과잉 허용**이 없다
- [ ] Public 접근은 **토큰 기반으로 row 단위 제한**한다(예: hold_token)
- [ ] anon UPDATE는 **토큰 일치 + 상태 조건** 둘 다 걸었다

### 2.2 실측 테스트(필수: “환경에서 실제로”)
- [ ] anon(토큰 없음) SELECT → **0 row 또는 403**
- [ ] anon(토큰 있음) SELECT → **해당 row만**
- [ ] anon UPDATE:
  - [ ] 토큰 불일치 → 0 row/403
  - [ ] 상태 불일치 → 0 row/403
  - [ ] 토큰+상태 일치 → 해당 row만 갱신

### 2.3 /api/health 보안 경계(해당 시)
- [ ] 외부 호출(토큰 없음) 응답이 **민감정보를 포함하지 않는다**
- [ ] 내부 호출(토큰 있음)에서만 상세 checks를 준다
- [ ] stack/error message/키 문자열 노출 없다

---

## 3) RPC 품질 체크 (RPC 변경 시 필수)

### 3.1 입력 검증/엣지
- [ ] 빈 배열/빈 jsonb 등 엣지를 명시적으로 거부한다(에러 메시지 포함)
- [ ] 타입/필수필드 누락 시 즉시 실패한다(부분 insert 금지)

### 3.2 락/동시성
- [ ] 경쟁 조건이 있는 테이블은 `FOR UPDATE`로 잠금한다
- [ ] **기대 건수 vs 잠금 건수**를 비교한다(부분 잠금이면 실패)
- [ ] 동일 입력 동시 2요청 시:
  - [ ] 1건만 성공 또는
  - [ ] 둘 다 “같은 결과”를 반환(멱등) 중 하나로 설계했다

### 3.3 원자성/트랜잭션
- [ ] RPC 내에서 “중간 상태”가 외부로 노출되지 않는다
- [ ] INSERT → 관계 테이블 INSERT → UPDATE 순서가 논리적으로 맞다
- [ ] 실패 시 롤백되어 **부분 데이터가 남지 않는다**

### 3.4 에러 메시지(운영/보안)
- [ ] 외부(Public)에서 내부 규칙/키/DB 구조가 추론 가능한 메시지를 내보내지 않는다
- [ ] 내부 로그/Sentry에는 원인 파악이 가능하도록 충분히 남긴다

---

## 4) 정산(Settlement) 전용 체크 (정산 로직 변경 시 필수)

### 4.1 정합성 불변조건(반드시 유지)
- [ ] settlement_items는 settlements와 1:N으로만 연결된다
- [ ] sold_items의 settlement_status 전이는 규칙대로만 바뀐다
- [ ] 중복 정산 방지(예: match_id UNIQUE 등)가 유지된다

### 4.2 계산 규칙
- [ ] COMMISSION_RATES 단일 소스(도메인 타입)만 사용한다
- [ ] 반올림/절사 규칙이 명시되어 있다(변경 시 영향 표기)
- [ ] 결과가 DB에 저장되기 전/후 동일함을 확인했다(증거 첨부)

### 4.3 “DB-우선 응답” 게이트(Rev.4 R4-03)
- [ ] 응답을 만들기 전에 DB write가 성공했는지 확인한다
- [ ] DB write 실패 시 **성공 응답이 나가지 않는다**
- [ ] 2번 호출해도 결과가 중복 생성되지 않는다(멱등/중복방지)

---

## 5) 업로드 세션 격리 체크 (upload_session_id 변경 시 필수)

- [ ] 세션 정책을 확정했다
  - [ ] 정책 1: 업로드 1회=1세션, 재시도만 동일 sessionId 사용
  - [ ] (옵션) 정책 2: 관리자별 last_session 매핑 테이블 운영
- [ ] deleteBySession(sessionId)는 **해당 세션 데이터만** 삭제한다
- [ ] 동시 업로드(A/B) 시 서로의 데이터가 절대 삭제되지 않는다
- [ ] 재시도(같은 sessionId) 시 중복/누락 없이 정리된다(증거 첨부)

## 6) 배치(_batch_progress) / partial / 재시도 체크 (해당 시 필수)

- [ ] _batch_progress DDL/CRUD 존재한다
- [ ] 429 등 부분 실패 발생 시:
  - [ ] status가 `partial`로 기록된다
  - [ ] failedIds가 남는다
  - [ ] completed/failed 카운트가 맞는다
- [ ] 재시도는 failedIds만 대상으로 가능하다(중복 처리 금지)
- [ ] batch_id UNIQUE로 중복 기록이 방지된다

## 7) 테스트 “명시” 체크 (L3는 필수)

### 7.1 RPC 통합 테스트(vitest)
- [ ] 빈 배열/누락 필드 → 실패
- [ ] 정상 입력 → 성공 + id 반환
- [ ] 동시 2요청 → 1건만 성공(또는 완전 멱등)

### 7.2 RLS 시나리오 테스트(가능하면 자동화, 아니면 수동 절차 + 증거)
- [ ] anon 토큰 없음 → 0 row/403
- [ ] 토큰 일치 → 해당 row만
- [ ] 업데이트 권한 경계가 지켜짐

### 7.3 E2E(정산/상태전이/고위험 라우트)
- [ ] CRITICAL 라우트 최소 1개는 E2E로 검증했다
- [ ] 실패 케이스(권한/충돌/검증 오류)를 포함했다

## 9) 최종 머지 전 “Fail 조건” (하나라도 해당하면 머지 금지)

- [ ] RLS가 과잉 허용(USING true 등) 상태다
- [ ] 동시성/락/멱등성에 대한 테스트나 증거가 없다
- [ ] DB 제약 추가인데 Preflight/정리 정책이 없다
- [ ] Public 라우트가 내부 정보(키/스택/규칙)를 노출한다
- [ ] 배치 partial 시 failedIds/상태 기록이 남지 않는다
