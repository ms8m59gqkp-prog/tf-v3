**적용 대상:**
L3 (DB/RPC/정산/상태전이/트랜잭션/동시성 변경)

> 전역 체크리스트(process-checklist1.md)를 통과한 상태에서 사용하는 "수술 전 체크리스트"다.

## 0. 적용 조건 확인

다음 중 하나라도 해당되면 이 문서를 적용한다:
- RPC 수정 또는 신규 작성
- 트랜잭션 코드 변경
- 정산 계산 로직 변경
- 상태전이 규칙 변경
- RLS 정책 변경
- UNIQUE/INDEX/CONSTRAINT 변경
- 배치 처리 로직 변경
- 업로드 세션 흐름 변경
- DB 결과가 달라지는 로직 변경

---

## 1. 데이터 정합성 검증

### 1.1 단일 실행 정상 케이스
- 정상 입력 시 기대 row 수 정확히 생성됨
- 기대 상태로 정확히 전이됨
- 계산 결과(금액/수량) 수동 계산과 일치

**증거:**
- 변경 전/후 DB 스냅샷
- SELECT 결과 캡처

### 1.2 엣지 케이스
- 빈 배열 입력 시 명확한 에러
- 이미 처리된 상태에서 재요청 시 CONFLICT 발생
- 존재하지 않는 ID 요청 시 NOT_FOUND 발생

---

## 2. 멱등성 검증 (필수)

### 2.1 동일 요청 2회 실행
- 1회 실행 → 정상
- 동일 요청 2회 → 추가 row 생성 없음
- 상태 2단계 이동 없음
- 정산 2건 생성 없음

**증거:**
- 실행 전 row 수
- 1회 후 row 수
- 2회 후 row 수

### 2.2 네트워크 재시도 시나리오
- 첫 요청 성공 직후 동일 요청 재실행
- 결과 동일
- DB 일관성 유지

---

## 3. 동시성 검증 (FOR UPDATE / RPC)

### 3.1 동시 요청 2개
- 동일 자원에 동시에 2개 요청
- 1개만 성공
- 1개는 명확한 에러 반환
- 데이터 꼬임 없음

### 3.2 레이스 컨디션 시나리오
- 상태 확인 후 업데이트 사이에 다른 요청 개입해도 정합성 유지
- `.eq('status', expected)` 패턴 적용 확인

---

## 4. RLS/보안 경계 검증

### 4.1 anon 접근
- 허용된 row만 조회 가능
- 토큰 없이 접근 시 0 row 또는 403
- 다른 사용자 데이터 접근 불가

### 4.2 update 제한
- 해당 row만 수정 가능
- 조건 미충족 시 update 0 row

### 4.3 민감 정보 노출 방지
- 에러 메시지에 내부 SQL/stack 없음
- API 키/토큰/비밀값 응답 없음

---

## 5. 정산 전용 추가 검증

### 5.1 계산 정확성
- 총매출 = 개별 합계
- 수수료율 정확
- 반올림 규칙 일관성 유지

### 5.2 중복 정산 방지
- 동일 sold_items 재정산 불가
- UNIQUE 제약 정상 작동

### 5.3 기간 겹침 방지
- 동일 판매자 기간 겹침 검증
- 중복 생성 불가

---

## 6. 업로드/세션 격리 (해당 시)

### 6.1 동시 관리자
- A/B 업로드 데이터 섞이지 않음
- 서로의 session_id 데이터 삭제 불가

### 6.2 동일 관리자 재시도
- 같은 session_id 재요청 시 중복 없음
- partial 상태 복구 가능

---

## 7. 배치/Partial 처리
- 429 발생 시 status = 'partial'
- failedIds 정확히 기록
- 재시도 시 failedIds만 재처리 가능
- _batch_progress row 업데이트 확인

---

## 8. 마이그레이션 전 Preflight (해당 시)
- 중복 탐지 쿼리 실행
- 고아 FK 탐지 쿼리 실행
- 정리 정책 명시
- UNIQUE 추가 전 수동 승인

---

## 9. 성능/제약 검증
- 목록 쿼리 .range() 적용
- 대량 .in() 호출 시 chunkArray 사용
- 인덱스 존재 확인
- EXPLAIN ANALYZE로 풀스캔 여부 확인 (필요 시)

---

## 10. 산출물 증거 (PR 필수 첨부)

PR에는 반드시 포함:
- DB 실행 전/후 비교 결과
- RPC 실행 로그
- 동시성 테스트 결과
- 멱등성 테스트 결과
- 영향 row 수 명시
