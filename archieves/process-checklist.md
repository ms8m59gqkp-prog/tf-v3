# process-checklist.md v1.0

**목적:**
감으로 "됐겠지"를 없애고,
모든 변경을 산출물 기반 검증으로 끝내기 위함.

---

## 0. 적용 범위

✅ 모든 코드 변경 PR에 무조건 적용 (예외 없음)
- UI 수정
- API 수정
- 서비스 로직 변경
- DB/RPC/RLS 변경
- 설정 변경
- 테스트 추가/수정

---

## 1. 공통 완료 기준 (모든 레벨 공통)

### 1.1 빌드/정적 검증
- tsc --noEmit 통과
- next build 성공
- ESLint 에러 0

### 1.2 테스트
- 변경 영향 코드에 대한 vitest run 실행
- 실패 테스트 0
- 기존 테스트 깨짐 없음

### 1.3 산출물 증거 보관

PR에 최소 1개 이상 포함:
- 빌드 로그
- 테스트 실행 로그
- API 응답 샘플
- DB 결과 스냅샷
- 워커/pm2 로그 일부

---

## 2. 변경 레벨 분류

변경 전에 반드시 레벨 지정.

### 🟢 L1 — UI/문구/스타일

로직 없음 / DB 영향 없음

**추가 체크:**
- 하드코딩된 API URL 없음
- 사진 경로 하드코딩 없음 (/uploads/, /public/ 직접 참조 금지)
- getPhotoUrl() 등 공용 헬퍼 사용

**증거:**
- 스크린샷 1장 또는 UI 동작 영상/이미지

### 🟡 L2 — API/서비스/비즈니스 로직

DB 구조 변경 없음
그러나 DB 결과에 영향 있음

**추가 체크:**
- 모든 API에 Zod 검증 존재
- requireAdmin 반환값 확인
- 서비스에서 NextRequest/NextResponse 직접 사용 없음
- 상태 전이 규칙 위반 없음
- `.or(\`` 사용 0건
- 목록 쿼리에 .range() 적용

**증거:**
- API 응답 JSON 샘플 1개
- DB 결과 스냅샷 1개 (변경 영향 확인)

### 🔴 L3 — DB/RPC/정산/상태전이 (고위험)

가장 엄격한 검증 필요

#### 2.3.1 DB 변경 여부 확인

다음 중 하나라도 해당되면 L3:
- 테이블/컬럼/인덱스 변경
- RLS 정책 변경
- RPC 수정
- 트랜잭션 코드 수정
- 정산 계산 로직 변경
- 상태전이 규칙 변경
- DB 결과가 달라지는 로직 변경

#### 2.3.2 멱등성(중복 방지) 검증

다음 영역은 반드시 동일 요청 2회 → 결과 동일 확인:
- 정산 생성
- 상태 전이
- 알림 발송
- Outbox
- 배치 처리

**체크:**
- 동일 요청 2회 실행
- 중복 row 생성 없음
- 상태 꼬임 없음

#### 2.3.3 RLS/보안 경계 검증
- anon 접근 시 허용된 row만 조회
- 토큰 없이 접근 시 0 row 또는 403
- update 시 해당 row만 수정 가능
- 민감 정보(API 키, 내부 메시지) 응답에 노출 없음

#### 2.3.4 동시성 검증 (RPC / FOR UPDATE)
- 동일 자원에 동시 요청 2개
- 1건만 성공
- 나머지 명확한 에러 반환
- 데이터 정합성 유지

#### 2.3.5 업로드 세션 격리 (해당 시)
- A/B 동시 업로드 데이터 섞이지 않음
- 같은 세션 재시도 시 중복 없음
- deleteBySession 동작 확인

#### 2.3.6 배치 Partial 처리
- 429 발생 시 partial 상태 기록
- failedIds DB 기록
- 재시도 시 failedIds만 재처리 가능

#### 2.3.7 DB 산출물 증거 (필수)

PR에 반드시 포함:
- 변경 전/후 DB row 비교
- RPC 실행 결과
- 영향 row 수 확인

## 3. Preflight (마이그레이션 전 필수)

DB 변경 시:
- 중복 탐지 쿼리 실행
- 고아 FK 탐지 쿼리 실행
- 정리 정책 명시
- 수동 승인 후 UNIQUE 제약 추가

---

## 4. 운영/성능 게이트

### 4.1 Supabase 제약
- 페이지네이션 limit ≤ 100
- 대량 .in() 호출 시 chunkArray 사용
- connection 과다 사용 없음
- SELECT * 남용 없음

### 4.2 Health Endpoint
- HEALTHCHECK_TOKEN 보호
- 외부 응답은 상세 정보 미노출
- 내부 모니터링 전용 상세 체크 분리

---

## 5. 에러 표준화

모든 API는 표준 구조 사용:
```json
{
  "success": false,
  "error": {
    "code": "CONFLICT | VALIDATION_ERROR | AUTH_REQUIRED | NOT_FOUND | INTERNAL",
    "message": "사용자 노출 메시지"
  }
}
```
- 내부 에러 메시지 직접 노출 금지
- Public 라우트는 무해화된 메시지 반환

---

## 6. PR 최종 승인 기준

PR은 다음 조건을 모두 만족해야 Merge 가능:
- 레벨 분류 명시 (L1/L2/L3)
- 공통 완료 기준 통과
- 레벨별 체크 통과
- 산출물 증거 첨부
- 리뷰어 1명 이상 승인
